# 头像昵称与定位授权优化设计

## 背景

当前首页同时启动头像昵称授权检查和定位请求。首次进入时，头像昵称弹层、系统定位弹窗、定位失败设置弹窗可能连续出现，容易打断用户。资料页头像更新只保存临时头像路径，没有上传到云存储；昵称保存失败时也缺少明确反馈。

## 目标

- 保留首次进入必须设置头像和昵称的规则，保证用户体系完整。
- 定位授权改为低打扰流程：首次进入可以展示默认地图，用户主动点击定位时再引导授权。
- 资料页头像更新上传至云存储后再保存，昵称更新有保存状态和失败反馈。
- 授权状态检查不闪屏，不在用户已授权时短暂显示授权弹层。

## 非目标

- 不调整数据库 schema。
- 不新增全局依赖。
- 不改变贴罚单、排行榜、路段详情的业务规则。
- 不接入新的地图 SDK 或定位服务。

## 方案

### 头像昵称授权

首页新增授权检查中状态。页面加载时先同步读取 `app.globalData.isAuthorized`；如果 app 仍在检查用户信息，则显示页面原有加载内容，不立即弹出授权层。只有确认未授权后才展示头像昵称弹层。

授权提交仍使用微信 `chooseAvatar` 和 `type="nickname"`。提交时上传头像到云存储，再调用 `user-info` 保存 `nickname` 和云存储 `fileID`。成功后更新 `app.globalData.userInfo`、`app.globalData.isAuthorized` 和页面状态，但不主动触发定位弹窗。

资料页复用同样的头像上传流程：选择头像后先上传，上传成功再保存 `fileID`。昵称输入后延迟保存，保存期间显示轻量状态；失败时重新拉取服务端用户信息，避免本地显示与数据库不一致。

### 定位授权

首页加载时尝试定位，但失败不弹设置弹窗，只展示默认坐标地图并记录定位状态。定位按钮成为明确入口：如果用户未拒绝过定位，点击后调用 `wx.getLocation`；如果用户已拒绝，点击后弹窗引导打开设置。

从 `wx.openSetting` 返回后，如果 `scope.userLocation` 已开启，则自动重新定位一次。定位成功更新地图中心；失败保持默认地图，并提示用户可以搜索路段名。

排行榜“我的城市”继续按需定位。用户切换到我的城市但没有城市信息时，触发同一套定位尝试；拒绝过定位时引导设置，仍失败时保持全国榜并提示。

## 数据流

1. `app.js` 启动：静默登录，调用 `user-info` 获取用户信息，写入 `globalData`。
2. 首页加载：等待授权检查完成，决定是否展示头像昵称弹层；地图可以先用默认坐标渲染。
3. 首次授权：选择头像、填写昵称、上传头像、保存用户信息、关闭弹层。
4. 主动定位：按钮触发权限判断，成功则更新地图中心，失败则保留默认地图。
5. 资料页更新：头像或昵称变更后保存到 `user-info`，成功同步全局用户信息。

## 错误处理

- 头像上传失败：提示“头像上传失败，请重试”，不提交用户信息。
- 昵称内容安全失败：沿用 `user-info` 返回错误提示。
- 用户信息保存失败：保留弹层或回滚资料页本地状态。
- 定位失败：不阻塞使用，提示可搜索路段名。
- 定位被拒绝：只在用户主动点击定位或切换我的城市时引导设置。

## 验证

- 首次进入未授权用户：不闪授权弹层，最终展示头像昵称弹层。
- 已授权用户：不展示头像昵称弹层，可直接搜索、点地图、贴罚单。
- 首次拒绝定位：页面仍显示默认地图，搜索可用。
- 点击定位且已拒绝：弹设置引导；开启权限后返回自动定位。
- 资料页更换头像：数据库保存云存储 `fileID`，不是临时路径。
- 资料页昵称保存失败：页面不长期停留在未保存状态。
